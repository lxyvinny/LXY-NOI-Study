# B3613 图的存储与出边的排序

## 题目描述

给定一个 $n$ 个点 $m$ 条边的有向图 $G$，结点编号从 $1$ 至 $n$。对于 $u = 1, 2, 3, \dots n$，依次完成如下要求：  
对于 $u$ 的所有出边（即从 $u$ 出发的边），按照**从小到大**的顺序输出出边所指向的节点编号。

**依次完成**的含义是，先按顺序输出 $u = 1$ 的出边所指向的点的编号，再按顺序输出 $u = 2$ 的出边所指向的点的编号……最后按顺序输出 $u = n$ 的出边所指向的点的编号。

## 输入格式

**本题单测试点内有多组数据**。

数据的第一行是一个整数 $T$，表示数据的组数。

对于每组数据的格式如下：  
每组数据的第一行是两个整数，分别表示点的个数 $n$ 和边的个数 $m$。  
接下来 $m$ 行，每行两个整数 $u, v$，表示一条由 $u$ 指向 $v$ 的边。

保证每组数据内不存在重边。

## 输出格式

对于每组数据：  
输出 $n$ 行，每行若干个用空格隔开的整数。第 $i$ 行输出节点 $i$ 的出边所指向的节点编号。

**注意，如果一个结点不存在出边，你同样需要输出一个空行**。

## 输入输出样例 #1

### 输入 #1

```
2
3 4
1 3
1 2
3 2
3 1
3 9
1 3
2 3
3 3
1 2
2 2
3 2
1 1
2 1
3 1
```

### 输出 #1

```
2 3

1 2
1 2 3
1 2 3
1 2 3
```

## 说明/提示

### 数据规模与约定：

对于全部的测试点，保证 $1 \leq T, n, m \leq 5 \times 10^5$，但同时各测试点的 $n$ 与 $m$ 之和均不超过 $5 \times 10^5$，即 $\sum n, \sum m \leq 5 \times 10^5$。且 $1 \leq u, v \leq n$，每组数据内不存在重边。

### 1. 什么是“图” (Graph)？
在数学里，**图**不是指我们画的一幅画，而是一种描述**关系**的工具。
* **点 (Nodes)**：想象成一个个小朋友。
* **边 (Edges)**：想象成小朋友之间连的线。
* **有向图 (Directed Graph)**：这意味着线是有方向的。就像水管里的水只能往一个方向流，或者你送礼物给别人，礼物是从你手里流向对方。



### 2. 数学中的“集合”与“排序”
这道题的核心数学操作是处理**集合 (Sets)**：
* **出边集合**：对于每个小朋友 $u$，他所有送出去的花构成了一个“收花人集合”。
* **从小到大排序**：数学上我们叫“升序排列”。
    > **例子**：如果 1 号送给了 $\{3, 1, 2\}$，我们要把它变成一个有序的序列 $(1, 2, 3)$。

---

### 3. “和”的约束（进阶数学概念）
题目中有一个看起来有点复杂的符号：$\sum n, \sum m \leq 5 \times 10^5$。
这里的 $\sum$ 是数学里的**求和符号**（念作 Sigma）。

* **它的意思**：虽然有很多组比赛（$T$ 组），但所有比赛里的小朋友总数加起来，以及所有花的总数加起来，都不会超过 50 万。
* **为什么重要？** 这告诉我们，我们不能在每组比赛开始时都准备一个“超级巨大”的本子（比如每次都开辟一个能装 50 万人的大空间），否则还没写几个字，本子就用光了。我们要**按需分配**空间。

---

### 4. 时间复杂度 (Time Complexity) —— 效率的数学
这是计算机科学里最重要的数学概念之一。
* 如果我们用最笨的方法，每找一个数字都把整本书翻一遍，那可能要花好几天。
* **排序的代价**：给 $k$ 个数字排序，数学上的效率通常是 $k \times \log(k)$。
* **总效率**：因为总边数 $m$ 是 50 万，我们要确保整理所有边的总时间能够在大约 1 秒钟内完成。

@利用题目中的样例数字示范题目计算过程。
为了让你更直观地看到这个过程，我们拿样例 1 的第一个小比赛（数据组）来当例子。

### 样例拆解：3 个小朋友，4 条小红花记录

**输入数据：**
* 小朋友总数 $n=3$，送花总数 $m=4$。
* 送花记录：`(1, 3)`, `(1, 2)`, `(3, 2)`, `(3, 1)`。

---

### 第一步：准备“小抽屉”
我们要为 1、2、3 号小朋友各准备一个空的文件夹。

| 小朋友编号 | 文件夹内容（初始为空） |
| :--- | :--- |
| **1 号** | `[ ]` |
| **2 号** | `[ ]` |
| **3 号** | `[ ]` |

---

### 第二步：收录名单（填写记录）
我们按照输入的顺序，把“收花人”放进对应的“送花人”文件夹里：

1.  读取 `1 3`：给 **1 号** 文件夹放入 `3`。
2.  读取 `1 2`：给 **1 号** 文件夹放入 `2`。
3.  读取 `3 2`：给 **3 号** 文件夹放入 `2`。
4.  读取 `3 1`：给 **3 号** 文件夹放入 `1`。

**收录完后的状态：**
* **1 号**：`[3, 2]`
* **2 号**：`[ ]`（没送过花，空的）
* **3 号**：`[2, 1]`

---

### 第三步：数学排序（从小到大）
现在我们要整理每个文件夹里的数字，让它们按顺序排好队：

* **1 号**：把 `[3, 2]` 变成 **`[2, 3]`**
* **2 号**：还是 **`[ ]`**
* **3 号**：把 `[2, 1]` 变成 **`[1, 2]`**



---

### 第四步：输出最后的结果
按照 1, 2, 3 的顺序把文件夹里的内容印出来：

* 第一行（1 号）：`2 3`
* 第二行（2 号）：（**空行**，因为没有花）
* 第三行（3 号）：`1 2`

**最终效果：**
```
2 3

1 2
```
这正是样例输出的前三行！

---

### 💡 为什么这么做最快？（数学小知识）

如果我们每读到一个送花记录就去翻书排序，会非常慢。
数学上，我们采用的是 **“先收集、后统一处理”** 的策略。

* **存储结构**：这种把每个点的邻居（收花人）存起来的方法，在数学和计算机里叫 **“邻接表” (Adjacency List)**。
* **总时间**：因为我们一共只处理了 $m$ 条边，最后排序的总工作量大约是 $m \times \log(\text{每个点的平均花数})$，这对于 50 万的数据量来说，计算机眨下眼就算完了。

你看，通过这种“小文件夹”的管理方式，是不是把复杂的图论题变成了一个简单的整理工作？
